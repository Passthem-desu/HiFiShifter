## Context

音高曲线在 Piano Roll 中的渲染涉及多个环节的坐标转换：

1. **后端数据提供**：`get_param_frames` 返回参数曲线数据，包含 `frame_period_ms`、`start_frame` 和数据数组
2. **前端数据请求**：`usePianoRollData.ts` 基于可见时间范围计算帧号范围和请求参数
3. **坐标转换渲染**：`render.ts` 中的 `drawCurveTimed` 将帧号转换为时间，再转换为像素坐标

**当前问题现象**：音高曲线出现"横向压缩"，即曲线的水平缩放比例不正确，导致数据点与时间轴不对应。

**代码现状**：
- 数据请求时间范围计算：`visibleStartSec` / `visibleDurSec` 基于 `scrollLeft` / `pxPerBeat` / `secPerBeat` 计算
- 帧号到时间转换：`tSec = (frame * framePeriodMs) / 1000`
- 时间到像素转换：`x = ((tSec - visibleStartSec) / visibleDurSec) * w`

**约束条件**：
- 必须保持与波形（waveform）渲染的时间轴一致性
- 支持任意缩放级别（`pxPerBeat` 可变）
- 支持滚动（`scrollLeft` 可变）
- 数据请求使用 margin 预加载以支持平滑滚动

## Goals / Non-Goals

**Goals:**
- 修复音高曲线的横向坐标转换，确保数据点与时间轴严格对应
- 验证并修正帧号-时间-像素的完整转换链路
- 添加调试日志以便验证转换的正确性
- 确保在不同缩放、滚动、BPM、采样率下渲染正确

**Non-Goals:**
- 不修改后端的数据格式和 API 接口
- 不改变 Piano Roll 的交互逻辑
- 不涉及参数编辑功能的修改
- 不优化渲染性能（除非发现明显问题）

## Decisions

### 决策 1：诊断方法 - 添加对齐验证点

**决策**：在渲染时添加已知参考点（如音频起始点、特定时间点）的可视化标记，以便直观验证对齐是否正确。

**理由**：
- 当前问题描述为"横向压缩"，但具体偏差量和模式未知
- 需要直观的视觉反馈来确认问题是否修复
- 调试日志已存在但可能不够直观

**备选方案**：
- 方案 A（采用）：在画布上绘制参考线（如每秒、每拍）和数据点的时间标记
- 方案 B：仅依赖控制台日志对比 - 不够直观，难以快速验证

### 决策 2：根因分析 - 重点检查三个疑点

基于代码审查，可能的问题点：

**疑点 1：`framePeriodMs` 语义理解偏差**
- 当前假设：`framePeriodMs` 是每帧的时间间隔（ms）
- 可能问题：如果后端返回的是采样率（Hz），需要转换为周期（ms）
- 验证方法：检查后端 `frame_period_ms` 的实际值和计算逻辑

**疑点 2：数据请求的量化（quantization）副作用**
- `usePianoRollData.ts` 中使用 `quantStepSec = 0.02` 对时间范围进行量化
- `paramStartSecQ` 和 `paramDurSecQ` 可能引入轻微偏差
- 影响：如果量化后的时间范围与实际帧号范围不匹配，会导致缩放误差

**疑点 3：`stride` 相关逻辑**
- 当前 `stride = 1`（全分辨率），但代码中预留了 stride 处理逻辑
- 如果后端实际返回的数据步进与前端假设不一致，会导致时间计算错误
- 验证方法：确认后端 `stride` 参数的实现与前端假设一致

**决策**：优先验证疑点 1 和疑点 2，因为它们最可能导致系统性的缩放偏差。

### 决策 3：修复策略 - 统一时间计算基准

**决策**：确保数据请求、数据缓存判断、渲染时使用完全相同的时间计算公式。

**具体措施**：
1. 提取公共函数 `framesToTime(frame, framePeriodMs)` 和 `timeToFrame(timeSec, framePeriodMs)`
2. 在所有涉及帧号-时间转换的地方使用统一函数
3. 确保 `paramCoversVisible` 的计算与渲染时的时间范围计算一致

**理由**：
- 当前代码中至少有 3 处独立计算帧号-时间转换（请求、缓存判断、渲染）
- 即使公式相同，手写多次容易出现细微差异（如舍入、边界处理）
- 统一函数可确保一致性并简化维护

**备选方案**：
- 方案 A（采用）：提取公共转换函数
- 方案 B：保持当前分散的计算但添加单元测试 - 难以保证运行时一致性

### 决策 4：参考对齐 - 波形作为真值基准

**决策**：将波形（waveform）的渲染作为正确性基准，确保音高曲线与波形在时间轴上严格对齐。

**理由**：
- 波形是直接基于音频采样的可视化，时间轴对应关系最直观
- 用户通常通过波形特征（如音符起始、结束）来判断音高曲线是否对齐
- 当前代码中波形渲染逻辑相对独立，可作为参考

**验证方法**：
1. 在波形特征点（如突变处）绘制垂直参考线2. 验证音高曲线的对应数据点是否在同一垂直线上
3. 在不同缩放级别重复验证

## Risks / Trade-offs

**[Risk 1] 修改坐标转换逻辑可能引入新的渲染问题**  
→ **Mitigation**: 
- 在修改前添加完整的调试日志，记录修改前后的关键计算值
- 保留原有逻辑代码（注释），便于对比和回滚
- 在多种场景下测试（不同 BPM、采样率、缩放级别）

**[Risk 2] 后端 `frame_period_ms` 语义理解错误导致修复方向偏离**  
→ **Mitigation**:
- 优先检查后端代码，确认 `frame_period_ms` 的实际计算逻辑
- 使用已知音频文件（如 1 秒 440Hz 正弦波）验证帧周期的正确性
- 添加前端校验，检测异常的 `framePeriodMs` 值（如 ≤0 或 >1000）

**[Risk 3] 量化逻辑（quantStepSec）导致边界情况下的微小误差**  
→ **Mitigation**:
- 量化步长（0.02s = 20ms）相对于典型帧周期（5ms）较大，考虑减小到 5ms
- 或者在计算 `startFrame` 时不使用量化后的时间，直接使用原始 `visibleStartSec`
- 添加边界测试（滚动到音频起始位置、结束位置）

**[Trade-off] 添加调试可视化会影响性能**  
→ **Mitigation**:
- 调试标记仅在开发模式或通过 localStorage 标志启用
- 使用简单的绘制操作（直线、文本），避免复杂计算
- 在生产构建中完全移除调试代码（通过条件编译）

## Migration Plan

1. **阶段 1：诊断和验证**（不修改核心逻辑）
   - 添加详细的调试日志和可视化标记
   - 确认问题的具体表现（缩放比例、偏移方向）
   - 验证后端 `frame_period_ms` 的语义和数值正确性

2. **阶段 2：修复核心逻辑**
   - 提取统一的帧号-时间转换函数
   - 修正发现的计算偏差
   - 调整或移除量化逻辑（if needed）

3. **阶段 3：验证和测试**
   - 在多种场景下测试渲染正确性
   - 对比波形对齐
   - 移除或条件化调试代码

**回滚策略**：
- 通过 Git 版本控制，每个阶段单独 commit
- 如果发现问题加重，立即回滚到前一个稳定状态
- 保留原有逻辑的注释代码，便于快速恢复

## Open Questions

1. **后端 `frame_period_ms` 的具体计算逻辑是什么？**
   - 需要查看 `state.rs` 中 `frame_period_ms()` 的实现
   - 确认是否与音频采样率、分析算法参数相关

2. **是否存在不同参数（pitch vs tension）使用不同帧周期的情况？**
   - 当前代码假设同一 track 的所有参数共享 `frame_period_ms`
   - 如果不同参数有不同周期，需要调整缓存和请求逻辑

3. **量化步长 0.02s 的选择依据是什么？**
   - 是否可以安全地减小或移除？
   - 对缓存命中率的影响如何？

4. **`stride` 参数未来是否会启用？**
   - 当前硬编码为 1，但代码中预留了处理逻辑
   - 如果未来启用，需要确保修复方案兼容
